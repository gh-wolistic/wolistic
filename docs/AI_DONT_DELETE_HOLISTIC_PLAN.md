# Holistic Plan Program

## Executive Summary For Founders

### Vision
Wolistic's core USP is an expert-backed holistic plan that combines exactly three expert roles (mind, body, diet), converts users from intent to payment, and continues into measurable outcomes on the dashboard.

### Current Reality
1. The top-of-funnel journey is partially built: landing search, results, expert review intake, and holistic plan pages exist.
2. Intake answers are persisted in backend.
3. Holistic plan suggestions are currently fallback/static and not generated by backend AI logic.
4. Auth continuation and holistic plan checkout flow are incomplete.

### Business Rules (Hard Constraints)
1. Every holistic plan must include one mind expert, one body expert, and one diet expert.
2. Holistic plan price is fixed at INR 5000 for current release.
3. All holistic plan code must be in dedicated, separate folders in frontend and backend.

### What Success Looks Like
1. User completes flow: search -> expert review -> holistic plan -> login (if needed) -> payment -> dashboard onboarding.
2. User sees real backend-generated plan options (not fallback).
3. Payment conversion happens on the selected plan and data is traceable end-to-end.
4. Experts can review, edit, and approve plan versions before user purchase where required.

### Top Risks
1. Drop-off due to broken auth continuation after Google sign-in.
2. Payment break because dedicated holistic plan checkout route is missing.
3. AI trust risk if recommendations are not explainable and triad constraints are violated.

### Founder-Level Milestones
1. Milestone A: Flow Stabilization (1 week)
2. Milestone B: Backend-Driven Holistic Plan (1-2 weeks)
3. Milestone C: AI Matching + Expert Review Loop (2-4 weeks)
4. Milestone D: Payment Completion + Dashboard Activation (1-2 weeks)
5. Milestone E: Launch Hardening and Metrics (ongoing)

## Engineering Implementation Spec For Developers

### 1) Scope and Boundaries

#### In Scope
1. Holistic plan recommendation pipeline.
2. Holistic plan APIs, models, and frontend pages.
3. Auth continuation context for proceed/request changes.
4. Razorpay payment initiation and verification for holistic plans.
5. Dashboard activation after successful payment.

#### Out of Scope (Current Phase)
1. Dynamic pricing engine.
2. Multi-currency support.
3. Full autonomous plan publishing without expert review controls.

### 2) Folder Separation Requirement (Mandatory)

#### Frontend
1. frontend/app/(public)/holistic-plan
2. frontend/components/holistic-plan
3. frontend/components/public/data/holistic-plan-api.ts
4. frontend/types/holistic-plan.ts
5. frontend/lib/holistic-plan

#### Backend
1. backend/app/api/routes/holistic_plan
2. backend/app/services/holistic_plan
3. backend/app/services/ai/holistic_plan
4. backend/app/schemas/holistic_plan.py
5. backend/app/models/holistic_plan

### 3) Current Known Status

#### Already Implemented
1. Search and navigation entry points.
2. Expert review intake persistence.
3. Booking payment infrastructure and provider abstraction.

#### Gaps To Close
1. Replace fallback plan data with backend plan APIs.
2. Add dedicated holistic payment page and route.
3. Respect continuation context after auth.
4. Wire holistic plan routes into backend router.

### 4) Data and Domain Model Requirements

#### Core Entities
1. holistic_plan_intake_session
2. holistic_plan_recommendation
3. holistic_plan_version
4. holistic_plan_expert_assignment
5. holistic_plan_review_event
6. holistic_plan_change_request
7. holistic_plan_purchase
8. user_active_holistic_plan

#### Status Model
1. draft
2. in_review
3. approved
4. published
5. purchased
6. active

#### Fixed Price Rule
1. Store canonical price in backend config/constant as 5000 INR.
2. Return price from API response for rendering.
3. Frontend must never be source of truth for plan price.

### 5) API Contracts (Proposed)

1. POST /api/v1/holistic-plan/intake
2. GET /api/v1/holistic-plan/recommendations?intake_id=...
3. GET /api/v1/holistic-plan/recommendations/{plan_id}
4. POST /api/v1/holistic-plan/recommendations/{plan_id}/request-changes
5. POST /api/v1/holistic-plan/recommendations/{plan_id}/checkout/order
6. POST /api/v1/holistic-plan/checkout/verify
7. POST /api/v1/holistic-plan/webhooks/razorpay
8. GET /api/v1/holistic-plan/my/active

### 6) Auth Continuation Contract

Use one consistent payload across all auth entry points:
1. returnTo
2. action
3. context

Example:
1. returnTo: /holistic-plan/plan-1?q=diet&scope=professionals
2. action: proceed_with_plan
3. context: { planId, q, scope, intent }

### 7) AI Architecture and Implementation

#### Where AI Is Implemented
1. backend/app/services/ai/holistic_plan/pipeline.py as orchestration entry point.
2. backend/app/services/ai/holistic_plan/retrieval.py for candidate retrieval.
3. backend/app/services/ai/holistic_plan/triad_matcher.py for role-constrained matching.
4. backend/app/services/ai/holistic_plan/scoring.py for explainable ranking.
5. backend/app/services/ai/holistic_plan/assembler.py for final plan payload.

#### Model Stack (Recommended)
1. Embeddings: BAAI/bge-small-en-v1.5 (or all-MiniLM-L6-v2 baseline).
2. Reranker: BAAI/bge-reranker-base.
3. Optional explanation generation: Mistral 7B Instruct or Llama 3.1 8B Instruct.

#### Public Datasets For Bootstrapping
1. MS MARCO (retrieval/reranking).
2. BEIR subsets (retrieval robustness).
3. MTEB tasks (embedding benchmarking).
4. STS-B (semantic similarity checks).

#### Non-Negotiable AI Constraints
1. Must enforce exactly one mind + one body + one diet expert in every recommendation.
2. Must provide explainable score breakdown fields in API.
3. Must support fallback deterministic rule-based ranking on model outage.

### 8) Detailed Prerequisites For AI Implementation

#### Product Prerequisites
1. Finalize triad taxonomy and role definition.
2. Define recommendation acceptance KPI and conversion KPI.
3. Define expert review SLA and approval policy.

#### Data Prerequisites
1. Clean expert profile metadata (specialization, availability, languages, service mode).
2. Structured intake schema with validated user answers.
3. Historical feedback schema for accepted/rejected/edited plans.
4. Explainability schema for score components.

#### Infrastructure Prerequisites
1. Environment setup for model inference (CPU baseline, optional GPU).
2. Vector retrieval layer (pgvector preferred for current stack).
3. Async job execution for recommendation generation.
4. Caching strategy for recent intake->recommendation requests.
5. Observability stack for latency, failures, and quality drift.

#### Backend Prerequisites
1. New migrations for holistic plan entities.
2. Strong API validation for continuation context and checkout payloads.
3. Idempotency keys for order creation and verify paths.
4. Signature verification for webhook processing.

#### Frontend Prerequisites
1. Dedicated holistic plan data layer and UI components.
2. Replace static fallback plan source in production path.
3. End-to-end context-preserving auth behavior.
4. Dedicated payment route and post-payment redirect behavior.

#### QA Prerequisites
1. Integration tests for recommendation pipeline.
2. E2E tests: anonymous -> login -> resume -> pay -> dashboard.
3. Failure scenario tests: model timeout, no candidates, payment failure.
4. Mobile UX test cases for entire funnel.

#### Security and Compliance Prerequisites
1. Redirect/returnTo strict allowlist and path validation.
2. Role-based access controls for expert review operations.
3. Audit trails for recommendation, edits, approvals, purchases.
4. Sensitive data handling policy for intake responses.

### 9) UX Requirements
1. Continuity UX: explicit messaging that user will resume exactly where they left off.
2. Plan confidence UX: why this plan, why these experts, expected timeline.
3. Request changes UX: first-class action with transparent turnaround expectation.
4. Payment UX: summary card with fixed INR 5000 and policy visibility.
5. Post-payment UX: immediate next steps and dashboard onboarding tasks.
6. Mobile-first quality across all steps.

### 10) Phase Plan
1. Phase 0 (1 week): stabilize flow, auth continuation, route guards, analytics.
2. Phase 1 (1-2 weeks): holistic plan schema/routes and frontend API wiring.
3. Phase 2 (2 weeks): AI retrieval + triad scoring + explainability.
4. Phase 3 (2 weeks): expert review workflow and plan versioning.
5. Phase 4 (1-2 weeks): checkout/payment completion + dashboard activation.
6. Phase 5 (1 week): UX polish and production hardening.
7. Phase 6 (ongoing): monitoring, reliability, and iterative model improvements.

### 11) Acceptance Criteria
1. Auth resume works from holistic plan proceed/request changes flow.
2. Holistic plan list and detail are backend-driven.
3. Fixed INR 5000 price is enforced server-side and displayed consistently.
4. Dedicated folder separation is respected across frontend/backend.
5. Successful payment activates dashboard plan state with next steps.
6. Full traceability exists for recommendation, edits, and approvals.
